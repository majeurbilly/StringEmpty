Phase 1 — Alignement API (.NET)

Une seule source de vérité — fin de json-server et des données simulées.

Avant — mock

// apiBaseUrl.ts
export const USE_MOCK_API = true;

// service
if (USE_MOCK_API) {
  return fetch("http://localhost:3001/items");
}

Après — API Crystal

// apiBaseUrl.ts
export const USE_MOCK_API = false;
export const API_URL = `${AUTH_URL}/api`;

// service
return apiClient.get(`${API_ITEMS_URL}`);
  • Commits clés : API Auteurs, champs ISBN/éditeurs, seed démo, support locationId
  • Domaines alignés : auth, inventaire, employés, congés, planning, paie, métriques RH

Phase 2 — Permissions dynamiques

Même modèle côté API et UI — configurable en base de données.

Avant — hybride

// Rôle Identity + presets locaux
if (user.role === "Admin") {
  showMenu("rh");
}
// json-server pour les rôles custom

Après — unifié

// Backend
[RequirePermission("read", "employee_profile")]

// Frontend (CASL)
GET /api/users/me/permissions
→ defineAbilityFor(user, permissions)
  • CRUD /api/roles + entités configurables
  • Labels de permissions dans l'UI admin
  • Permissions inventaire scopées par succursale (location-scoped)

Phase 3 — Scoping employé

Chaque employé ne voit que ses données RH.

  • EmployeeScopeService — filtre backend centralisé
  • Congés, quarts, pointages, feuilles de temps scopés au profil connecté
  • Portail /mon-espace — 5 onglets (horaire, congés, fiche, pointages, feuille)
  • GET /api/employee-profiles/me pour la fiche personnelle
  • Inventaire limité à la succursale de l'utilisateur
Règle : sans manage:{entité}, l'API filtre automatiquement sur le EmployeeProfileId lié au compte.

Flux de pointage

  • Widget punch-in / punch-out sur le dashboard
  • PunchEligibilityService — validation du quart lié à l'employé
  • BusinessClock — règles métier (fuseau America/Toronto)
  • Gestion du cas too_late pour un punch tardif

Rôles unifiés — DynamicRoleId

  • Un seul champ sur l'utilisateur : DynamicRoleId
  • 4 presets (Admin, Gérant, Assistant, Employé) + rôles personnalisés
  • Suppression du code Identity legacy redondant
  • Admin peut créer et assigner un rôle depuis l'UI
  • Types de compte clarifiés sur la fiche utilisateur

Refactor RH (juin) : retrait des rôles organisationnels du modèle de sécurité, ajout des périodes de paie et wizards d'onboarding employé.

Défis & solutions

Défis rencontrés

  • • Synchroniser permissions backend et boutons frontend
  • • Migrer sans casser les tests d'intégration (SQLite partagé)
  • • Unifier deux systèmes de rôles (Identity + dynamique)
  • • Préparer le déploiement (URLs, port API 8080)
  • • Nettoyer le code legacy en fin de sprint

Solutions appliquées

  • • GET /api/users/me/permissions comme source unique pour CASL
  • • Tests backend avec utilisateurs jetables en mémoire
  • • Migration vers DynamicRoleId (PR 2307)
  • • Config production + URLs relatives API
  • • Refacto fichiers orphelins + suppression sujets legacy
