L'ERP Crystal — c'est quoi ?

Crystal ERP est un système de gestion intégré pour une librairie multi-succursales. Il centralise l'inventaire (catalogue, stock par succursale) et les ressources humaines (employés, horaires, congés, paie) dans une seule application web.

Problème résolu

Avant le sprint 4, le frontend utilisait parfois des données simulées (json-server) et les permissions n'étaient pas alignées entre l'API et l'interface. Aujourd'hui, une seule API .NET fait foi : chaque écran et chaque bouton respecte les mêmes règles de sécurité que le backend.

Parcours type d'une journée

  1. Connexion — l'utilisateur s'authentifie sur /login ; l'API renvoie un JWT.
  2. Dashboard — widgets adaptés au rôle (prochain quart, congés en attente, métriques RH…).
  3. Travail — navigation via la sidebar : inventaire, RH gestionnaire ou Mon espace employé.
  4. Actions — créer un congé, approuver une absence, ajuster un stock, gérer un employé… selon les permissions.
  5. Données filtrées — un employé ne voit que ses congés, quarts et pointages ; un gérant voit toute l'équipe.
Deux « rôles » distincts : le rôle applicatif (Admin, Gérant, Assistant, Employé) contrôle la sécurité et les menus. Le rôle organisationnel RH (Ventes, Logistique…) est une donnée métier dans les référentiels RH.

Comment ça fonctionne

Le système applique les droits à trois niveaux : route frontend, interface (boutons/menus) et API backend.

1. Auth
POST /api/auth/login → JWT (rôle Identity : Admin, Gerant, Assistant, Employee)
2. Permissions
GET /api/users/me/permissions → liste { action, subject } (ex. read:leave_request)
Si l'utilisateur a un dynamicRoleId, les permissions du rôle dynamique remplacent le preset.
3. Frontend
CASL (defineAbilityFor) → sidebar, widgets dashboard, boutons Ajouter/Modifier/Supprimer masqués si non autorisé.
Routes /rh/* et /ir bloquées sans read:hr_dashboard (redirection vers le dashboard).
4. Backend
[Authorize] + RequirePermission sur chaque endpoint.
EmployeeScopeService : sans manage:{subject}, filtrage automatique sur le profil RH de l'utilisateur connecté.

Règles de permission

RègleEffet
manage:allAccès complet (Admin uniquement)
manage:{entité}CRUD complet + voir toutes les données de l'entité
read/create/update/delete:{entité}Action précise sur l'entité
Pas de manage + profil RH liéScoping « mes données » (congés, quarts, pointages, feuilles)

Les 4 rôles applicatifs

Chaque compte test est lié à un rôle Identity. Les permissions par défaut sont définies dans defaultRolePermissions.ts (backend + frontend synchronisés via l'API).

🔴 Administrateur

Contrôle total du système. Crée des utilisateurs, des rôles dynamiques personnalisés, gère tout l'inventaire et tout le module RH. Seul rôle avec manage:all.

🔵 Gérant de succursale

Pilote l'activité quotidienne : approuve les congés, consulte les métriques RH, gère les employés et le stock. Pas d'administration des rôles système ni création d'utilisateurs.

🟣 Assistant

Appui opérationnel : accès au menu RH en lecture (sauf création de congés et pointages pour soi), consultation du catalogue et de l'inventaire. Pas d'accès à la paie ni à la gestion des utilisateurs.

🟢 Employé

Utilisateur terrain : pas de menu RH gestionnaire. Accède à Mon espace pour son horaire, ses congés et sa fiche. Peut créer des demandes de congé et des pointages pour lui-même uniquement.

Ce que l'employé ne peut pas faire

  • Accéder au menu RH gestionnaire (/rh/employes, paie, planning global…) — redirection dashboard
  • Lister tous les employés via GET /api/employee-profiles — 403 Forbidden
  • Voir les congés, quarts ou pointages d'un collègue — 404 ou liste filtrée
  • Modifier le stock, créer des articles, gérer la paie ou les rôles système
  • Approuver ou rejeter des demandes de congé (réservé au gérant/admin)

Ce que le gérant ne peut pas faire

  • Créer ou supprimer des rôles dynamiques (manage:user_role — Admin seulement)
  • Créer des utilisateurs via l'UI (pas de permission create:user — Admin via manage:all)

Démarrer le projet

Option A — Reset complet (./start.sh)

Attention — reset nucléaire : ce script supprime les volumes Docker, efface les migrations EF Core, régénère une migration InitialCreate et reconstruit tout sans cache. À utiliser pour une base 100 % vierge (première install ou reset total).

Prérequis : Docker, Node.js/npm, pnpm, SDK .NET 9.

# Linux, macOS ou WSL — à la racine du dépôt
chmod +x start.sh
./start.sh

Ce que fait start.sh :

  1. Vérifie npm, pnpm, dotnet et installe dotnet-ef si besoin
  2. docker-compose down -v — purge conteneurs + volumes (DB à neuf)
  3. Supprime les anciennes migrations et crée InitialCreate
  4. pnpm install dans frontend/
  5. docker-compose build --no-cache puis up -d
  6. Affiche les logs (docker-compose logs -f)

Option B — Démarrage quotidien (sans reset)

Pour le développement courant, sans effacer la base :

docker compose up -d --build

Option C — Arrêter la stack

docker compose down          # garde les volumes
docker compose down -v       # efface aussi la base
Une fois démarré : ouvrez localhost:3000 et connectez-vous avec un compte test (voir section Comptes).

Environnement Nix + WSL

Le fichier flake.nix à la racine fournit un shell reproductible (dotnet 9, Node 22, pnpm, Docker) et des commandes nix run pour tester et lancer E2E.

Prérequis WSL

  1. Nix installé dans votre distro WSL (flakes activés)
  2. Docker Desktop lancé sur Windows
  3. Settings → Resources → WSL Integration → activer votre distro
  4. Optionnel : vérifier docker info dans WSL

Entrer dans le shell Nix

# Depuis la racine du projet (chemin WSL, ex. /mnt/c/Users/.../ERP)
nix develop

Le shell affiche les versions dotnet/node/pnpm et l'état Docker. Si Docker est indisponible, il rappelle d'activer l'intégration WSL.

Workflow recommandé sous WSL

# 1. Shell reproductible
nix develop

# 2. Démarrer la stack et attendre qu'elle réponde
nix run .#docker-up-wait

# 3. Tests unitaires / intégration
nix run .#backend-test
nix run .#frontend-test

# 4. E2E (première fois : installer Chromium)
nix run .#e2e-install
nix run .#e2e-test

# Ou tout en une commande :
nix run .#verify
e2e-test vérifie que http://localhost:3000 répond. Si la stack n'est pas démarrée, lancez d'abord nix run .#docker-up-wait.
